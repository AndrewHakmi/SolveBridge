## 1. Концепция и цели

SolveBridge — это **AI-Native Talent & Knowledge Operating System** (ОС Таланта и Знаний). Платформа соединяет компании, студентов, менторов и образовательные организации вокруг реальных задач.

**Четыре ключевых принципа:**

| Принцип | Что это значит |
|---|---|
| **Цифровые артефакты** | Каждый результат работы фиксируется как неизменяемый объект с хэшем содержимого. Не «был проект», а «вот доказательство». |
| **Граф компетенций** | Навыки не декларируются — они выводятся из артефактов. Каждый навык должен быть подкреплён конкретным доказательством. |
| **Proof of Mentorship** | Менторская активность логируется в блок-образные записи с весом сложности, что делает наставничество измеримым и монетизируемым. |
| **360° Оценка** | Качество проекта формируется из четырёх независимых источников: ментор, клиент, коллеги, артефакты. |

**Проблемы, которые решает платформа:**
- Студент не может доказать работодателю реальный опыт — только диплом.
- Компания не может найти студента по реальным навыкам (не по строчкам в резюме).
- Ментор работает «в тени» без учёта и вознаграждения.
- Компании обходят платформу, напрямую нанимая найденных через неё людей.

## 2. Участники и роли

Платформа знает **7 ролей**. Роль хранится в поле `role` модели `User` и определяет, что пользователь видит и может делать в UI.

### 2.1 Таблица ролей

| Роль | Код | Русское название | Что делает |
|---|---|---|---|
| Клиент | `client` | Заказчик | Создаёт задачи, выбирает исполнителей, оценивает результат |
| Студент | `student` | Студент | Откликается на задачи, проходит верификацию в вузе |
| Исполнитель | `executor` | Исполнитель | Профессионал, берёт задачи; аналог студента, но без вузовой верификации |
| Ментор | `mentor` | Ментор | Сопровождает исполнителей, ставит печать на артефакты, логирует активность |
| Администратор | `admin` | Администратор | Управляет партнёрами, видит всю статистику, модерирует |
| Партнёр | `partner` | Партнёр | Представитель вуза, технопарка или госоргана; управляет компаниями и менторами |
| Компания | `company` | Компания | Организация-заказчик, добавленная партнёром; создаёт задачи, нанимает |

## 3. Организации и структура экосистемы

Организации — это юридические или институциональные сущности, к которым привязаны задачи и партнёры.

### 3.1 Типы организаций

| Тип | Код | Примеры | Роль в платформе |
|---|---|---|---|
| Компания | `company` | ООО Татнефть, ИП Иванов | Создаёт задачи, является нанимателем |
| Университет | `university` | КФУ, Иннополис, КНИТУ | Верифицирует студентов, выдаёт менторов |
| Технопарк | `infrastructure` | Иннополис ОЭЗ | Партнёр-инфраструктура, поддерживает экосистему |
| Гос. орган | `government` | Минцифры РТ | Регулятор/партнёр, может добавлять компании |

### 3.2 Иерархия организаций

```
Администратор платформы
    │
    ├─► Вуз-партнёр (university)
    │       ├─► Верифицирует студентов
    │       ├─► Регистрирует менторов
    │       └─► Добавляет компании-партнёры
    │
    ├─► Технопарк (infrastructure)
    │       └─► Добавляет компании-партнёры
    │
    └─► Компания (company)
            ├─► Создаёт задачи
            └─► Выдаёт исполнителям задачи
```

### 3.3 Связь задачи с организацией

Каждая задача обязательно принадлежит организации (`organization_id` — FK на `organizations.id`). Это позволяет системе анти-дизинтермедиации отслеживать пары «компания–исполнитель».

## 4. Ключевые доменные концепции

### 4.1 Цифровой артефакт (Digital Artifact)

**Артефакт** — это атомарный, неизменяемый результат работы. Он является доказательством, а не просто файлом.

**Поля артефакта:**

| Поле | Тип | Смысл |
|---|---|---|
| `content_hash` | SHA-256 | Криптографический отпечаток содержимого |
| `reusability_index` | 0.0–1.0 | Насколько артефакт пригоден для повторного использования другими |
| `mentorship_seal` | bool | Ментор проверил и одобрил этот артефакт |
| `type_id` | FK→artifact_types | Тип: код, отчёт, презентация, дизайн, тест и т.д. |
| `git_url` | string | Ссылка на коммит/репозиторий |
| `metadata` | JSONB | Произвольные метаданные (размер, язык, библиотеки) |
| `project_id` | FK | Проект, к которому относится |
| `owner_team_id` | FK | Команда-владелец |

**Жизненный цикл артефакта:**
1. Создаётся при сдаче работы или через Git-интеграцию
2. Ментор проверяет и при подтверждении ставит `mentorship_seal = true`
3. Артефакт становится доказательством для обновления Графа компетенций

### 4.2 Граф компетенций (Capability Graph)

Граф — это база знаний о навыках. Каждый узел (`CapabilityEdge`) — доказанный навык.

**Ключевой инвариант:** навык без артефакта существовать не может.
Поле `evidence_artifact_id` — обязательная FK на артефакт.

**Уровни владения (`proficiency_level`):**
- `0.0–0.3` — базовое знакомство
- `0.3–0.6` — рабочий уровень
- `0.6–0.85` — уверенное владение
- `0.85–1.0` — экспертиза

Одна запись в графе = тройка `(entity_type, entity_id, skill_id)`.
- `entity_type` = `"user"` (навыки пользователя) или `"team"` (навыки команды)
- `entity_id` = UUID пользователя или команды

При повторной загрузке того же навыка запись **обновляется** (upsert), а не дублируется.

### 4.3 Proof of Mentorship (PoM)

**PoM** — это лог менторских действий, из которого платформа вычисляет ценность ментора.

**Типы действий (`action_type`):**
- `code_review` — ревью кода
- `session` — сессия 1:1
- `feedback` — письменная обратная связь
- `workshop` — групповой воркшоп
- `artifact_review` — проверка артефакта

**Формула PoM-баллов:**
```
PoM_score = sum(duration_minutes × complexity_weight)
```

`complexity_weight` — множитель от 0.5 до 3.0, задаётся вручную при логировании действия (насколько сложной была задача, над которой работали).

### 4.4 360° Оценка

Качество проекта (`success_rate`) вычисляется взвешенным средним четырёх источников:

```python
weights = {
    mentor:   0.40,   # 40% — оценка ментора (самый важный сигнал)
    client:   0.30,   # 30% — оценка заказчика
    peer:     0.20,   # 20% — оценка коллег
    artifact: 0.10,   # 10% — автоматическая оценка артефактов
}
```

Если некоторые оценки отсутствуют, вес нормируется на имеющиеся:
```python
total_weight = sum(present weights)
success_rate = sum(score × weight) / total_weight
```

Пример: если есть только mentor (9/10) и client (7/10):
```
success_rate = (9×0.40 + 7×0.30) / (0.40 + 0.30) = (3.6 + 2.1) / 0.70 = 5.7 / 0.70 ≈ 8.14
```

### 4.5 Анти-дизинтермедиация

Защита платформы от прямого найма исполнителей в обход системы.

**Правило:** первые **3 задачи** между парой «компания–исполнитель» обязательно должны выполняться с ментором. После 3 успешно завершённых задач правило деактивируется и ментор перестаёт быть обязательным.

**Логика активации:**
1. При назначении исполнителя система ищет запись `AntiDisintermediationRule` с ключом `(company_org_id, executor_id)`.
2. Если записи нет — создаётся с `required_task_count=3, completed_task_count=0, is_active=true`.
3. Если `is_active=true AND mentor_required=true AND completed_task_count < 3` — назначение без ментора блокируется с ошибкой 409.
4. При переводе задачи в `completed` — `completed_task_count` инкрементируется.
5. Когда `completed_task_count >= required_task_count` — `is_active` автоматически становится `false`.

## 5. Жизненный цикл задачи (BMC Marketplace)

### 5.1 Статусы задачи

```
open → assigned → completed
  ↓                  
disputed ─────────────→ completed / canceled
  ↓
canceled
```

| Статус | Описание |
|---|---|
| `open` | Задача опубликована, принимаются отклики |
| `assigned` | Выбран исполнитель, работа идёт |
| `disputed` | Открыт диспут, работа приостановлена |
| `completed` | Задача завершена, счётчик анти-дизинтермедиации растёт |
| `canceled` | Задача отменена |

### 5.2 Полный жизненный цикл задачи

#### Шаг 1: Создание задачи

**Кто:** Клиент или Администратор или Компания.

**Поля:**
- `organization_id` — обязательно, FK на организацию-компанию
- `title` — название (до 300 символов)
- `description` — полное описание
- `category` — категория (default: `general`)
- `budget_amount_rub` — бюджет в рублях (необязательно)
- `required_skills` — массив кодов навыков (JSONB)
- `due_at` — дедлайн (необязательно)

Задача создаётся в статусе `open`.

#### Шаг 2: Отклик исполнителя

**Кто:** Студент или Исполнитель.
**Условие:** задача в статусе `open`.

**Поля отклика:**
- `applicant_id` — ID откликающегося
- `proposed_amount_rub` — предложенная сумма (необязательно)
- `message` — сопроводительное письмо

**Особенность:** повторный отклик от того же пользователя **обновляет** существующий (upsert по `(task_id, applicant_id)`), не создаёт дубликат.

#### Шаг 3: Просмотр откликов клиентом

Клиент видит список откликов с:
- Именем/email кандидата
- Предложенной суммой
- Сообщением
- **Статусом верификации в вузе** — если кандидат является студентом и прошёл верификацию, рядом отображается бейдж с названием вуза (зелёный = подтверждён, синий = на проверке).

#### Шаг 4: Назначение исполнителя

**Кто:** Клиент или Администратор.

Перед назначением система выполняет **три последовательные проверки**:

**Проверка 1 — Compliance:**
```
SELECT ComplianceProfile WHERE user_id = executor_id
```
- Если профиль не найден → 409 «Исполнитель не дал согласие на ПДн»
- Если `pdn_consent = false` → 409 «Нет согласия 152-ФЗ»
- Если `npd_status NOT IN ('verified', 'not_required')` → 409 «Не подтверждён статус самозанятого (422-ФЗ)»

**Проверка 2 — Анти-дизинтермедиация:**
```
SELECT AntiDisintermediationRule WHERE company_org_id = task.organization_id AND executor_id = payload.executor_id
```
- Если правила нет — создаётся новое с требованием ментора
- Если `is_active AND mentor_required AND completed < 3 AND mentor_id = null` → 409 «Нужен ментор на первые 3 задачи»

**Проверка 3 — Состояние задачи:**
- Задача должна быть в `open` или `assigned`

Если все проверки прошли → создаётся `TaskAssignment`, задача переходит в `assigned`.

#### Шаг 5: Выполнение работы

На этом этапе исполнитель работает, ментор сопровождает.

Ментор может:
- Логировать активность (`POST /api/mentor-activity`)
- Ставить печать на артефакты (`PATCH /api/artifacts/{id}` с `mentorship_seal=true`)
- Обновлять граф компетенций исполнителя

#### Шаг 6: Завершение задачи

**Кто:** Клиент или Администратор.

`POST /api/tasks/{task_id}/status` с `{"status": "completed"}`

При завершении:
1. Задача переходит в `completed`
2. Система находит `TaskAssignment`
3. Находит `AntiDisintermediationRule` для этой пары
4. `completed_task_count += 1`
5. Если `completed_task_count >= required_task_count` → `is_active = false` (ментор больше не обязателен)

## 6. Верификация студентов

Верификация подтверждает принадлежность студента к конкретному вузу-партнёру.

### 6.1 Зачем нужна верификация

- Клиент видит, что кандидат — реальный студент (а не самозванец)
- Вуз получает данные о трудоустройстве своих студентов
- Платформа может предлагать задачи, требующие студенческого статуса

### 6.2 Поток верификации

```
Студент                    Вуз-партнёр              Клиент
   │                            │                      │
   ├─ Открывает Профиль         │                      │
   ├─ Выбирает вуз              │                      │
   ├─ Вводит номер студбилета   │                      │
   ├─ POST /verifications/students                      │
   │    status = "pending"      │                      │
   │                            │                      │
   │              ◄─── Кабинет партнёра показывает заявки
   │                            ├─ PATCH /verifications/{id}/review
   │                            │   action = "approve" / "reject"
   │                            │                      │
   ◄── status → "approved" ─────┤                      │
                                                       │
                            Клиент видит бейдж ────────►
                            "КФУ ✓" рядом с кандидатом
```

### 6.3 Состояния верификации

| Статус | Описание |
|---|---|
| `pending` | Заявка подана, ждёт решения вуза |
| `approved` | Вуз подтвердил — студент верифицирован |
| `rejected` | Вуз отклонил (документ не совпадает, студент отчислен и т.д.) |

**Правила:**
- Студент не может подать повторную заявку в тот же вуз, пока предыдущая в `pending`
- После одного `approved` форма повторной подачи скрывается
- Можно быть верифицированным сразу в нескольких вузах

### 6.4 Отображение для клиента

На странице откликов к задаче клиент видит:
- Зелёный бейдж (ShieldCheck) + название вуза → `approved`
- Синий бейдж (GraduationCap) + «· ожидает» → `pending`
- Текст «Вуз не указан» → верификаций нет

## 7. Compliance-профиль (152-ФЗ и 422-ФЗ)

Compliance-профиль — обязательное условие для назначения исполнителя на задачу.

### 7.1 Поля профиля

| Поле | Значения | Смысл |
|---|---|---|
| `pdn_consent` | `true / false` | Согласие на обработку персональных данных (152-ФЗ) |
| `pdn_consent_at` | datetime | Когда дано согласие |
| `npd_status` | `unknown, pending, verified, not_required` | Статус самозанятого (НПД, 422-ФЗ) |
| `npd_verified_at` | datetime | Когда подтверждён статус |
| `metadata` | JSONB | Дополнительные данные (ИНН, дата регистрации и т.д.) |

### 7.2 Логика проверки при назначении

```python
_NPD_OK = {"verified", "not_required"}

if not compliance or not compliance.pdn_consent:
    raise 409 "Нет согласия на ПДн (152-ФЗ)"

if compliance.npd_status not in _NPD_OK:
    raise 409 "Не подтверждён статус самозанятого (422-ФЗ)"
```

### 7.3 Как студент заполняет compliance

**Путь:** Профиль → Документы и согласия

1. Ставит галочку «Согласие на обработку персональных данных»
2. Выбирает статус НПД из выпадающего списка
3. Нажимает «Сохранить» → `PUT /api/compliance/profiles`

**Подсказки в UI при ошибке назначения:**
- Ошибка 152 → «Исполнитель должен открыть Профиль → Документы и включить согласие»
- Ошибка 422/самозанятого → «Исполнитель должен указать статус НПД»
- Ошибка ментора → «Для первых 3 задач нужно указать mentor_id»

## 8. Анти-дизинтермедиация

Подробно — см. п. 4.5. Здесь приводится полная таблица состояний.

### 8.1 Состояния правила

| `is_active` | `completed_task_count` | `required_task_count` | Ментор нужен? |
|---|---|---|---|
| `true` | 0 | 3 | Да (первая задача) |
| `true` | 1 | 3 | Да (вторая задача) |
| `true` | 2 | 3 | Да (третья задача) |
| `false` | 3 | 3 | Нет — правило выключено |
| `false` | любое | любое | Нет — правило выключено |

### 8.2 Что происходит, если ментора нет

Клиент получает ошибку 409:
```
"Для первых 3 задач пары компания–исполнитель требуется ментор (защита от обхода платформы)"
```

В UI появляется подсказка:
> **Нужен ментор на первые задачи**  
> Для первых 3 задач пары компания–исполнитель нужно указать mentor_id при назначении.

Клиент может прямо в форме назначения:
- Выбрать ментора из выпадающего списка
- Нажать «+ Добавить ментора» и выбрать пользователя из системы

## 9. Граф компетенций (Capability Graph)

### 9.1 Как работает upsert

```
PUT /api/capability
{
  "entity_type": "user",
  "entity_id": "uuid-исполнителя",
  "skill_code": "python",
  "skill_name": "Python",
  "proficiency_level": 0.75,
  "evidence_artifact_id": "uuid-артефакта"
}
```

Система:
1. Ищет Skill по `code` → если не найден, создаёт новый
2. Выполняет `INSERT ... ON CONFLICT DO UPDATE` по `(entity_type, entity_id, skill_id)`
3. Обновляет `proficiency_level` и `evidence_artifact_id`

### 9.2 Фильтрация

```
GET /api/capability?entity_type=user&entity_id={uuid}
```

Возвращает все навыки пользователя с уровнями и ссылками на доказательства.

### 9.3 Применение в платформе

- **Talent Directory** — поиск исполнителей по навыкам
- **Task matching** — сопоставление `required_skills` задачи с графом кандидата
- **Портфолио студента** — граф как публичный профиль компетенций
- **Аналитика вуза** — какие навыки воспитывают студенты

## 10. Цифровые артефакты

### 10.1 Типы артефактов (`artifact_types`)

Хранятся в таблице `artifact_types` с полями `code` и `name`. Примеры типов:

| Код | Название |
|---|---|
| `code` | Исходный код |
| `report` | Отчёт/документация |
| `design` | Дизайн/прототип |
| `presentation` | Презентация |
| `test` | Тесты/QA |
| `dataset` | Набор данных |

### 10.2 Reusability Index

`reusability_index` (0.0–1.0) показывает, насколько артефакт пригоден для использования в других проектах:

- `0.0` — специфический результат, бесполезен вне контекста
- `0.5` — типовое решение, требует адаптации
- `1.0` — универсальный компонент, готов к реиспользованию

Влияет на `artifact_score` при 360° оценке.

### 10.3 Mentorship Seal

Когда ментор проверил и одобрил результат:
```
PATCH /api/artifacts/{id}
{ "mentorship_seal": true }
```

Это действие:
1. Повышает доверие к артефакту
2. Является основанием для выставления высокого `mentor_score` проекту
3. Может обновить `proficiency_level` в графе компетенций

## 12. 360° Оценка проекта

### 12.1 Установка оценок

```
POST /api/scoring/projects
{
  "project_id": "uuid",
  "mentor_score": 8.5,    // 0–10
  "client_score": 7.0,
  "peer_score": 8.0,
  "artifact_score": 9.0
}
```

Оценки обновляются инкрементально — можно поставить сначала mentor, потом добавить client.

### 12.2 Вычисление success_rate

```python
# weights: mentor=0.40, client=0.30, peer=0.20, artifact=0.10
# если не все оценки есть — нормируем на присутствующие

total_weight = sum(w for score, w in present_parts)
success_rate = sum(score × w for score, w in present_parts) / total_weight
```

Результат хранится в поле `success_rate` модели `Project`.

### 12.3 Кто ставит какую оценку

| Оценка | Кто ставит | Когда |
|---|---|---|
| `mentor_score` | Ментор | По завершении работы |
| `client_score` | Клиент | После приёмки результата |
| `peer_score` | Коллеги по команде | После code review |
| `artifact_score` | Автоматически/ментор | Исходя из reusability_index артефактов |

## 15. Диспуты (SLA 48ч)

### 15.1 Открытие диспута

**Кто:** Любой авторизованный пользователь (студент, клиент, ментор).

```
POST /api/tasks/{task_id}/disputes
{
  "opened_by_id": "uuid",
  "reason": "Результат не соответствует ТЗ: ожидалось X, получили Y",
  "sla_minutes": 2880   // default 48ч = 2880 минут
}
```

**При создании диспута:**
1. Создаётся запись `Dispute` со статусом `open`
2. `sla_deadline = now + sla_minutes`
3. Задача переходит в статус `disputed` (если не была `completed`/`canceled`)

### 15.2 Состояния диспута

| Статус | Описание |
|---|---|
| `open` | Диспут открыт, идёт разбор |
| `resolved` | Диспут разрешён |

### 15.3 Отображение SLA

В UI рядом с диспутом показывается:
> SLA до: 31.05.2026, 14:35

Администратор должен разрешить диспут до истечения дедлайна.

### 15.4 Кто разрешает диспут

В текущей версии: Администратор через изменение статуса задачи и диспута.
Логика разрешения (кому перечислить деньги, как обновить score) — вне MVP.

## 16. Git-интеграция

### 16.1 Webhook от системы контроля версий

При каждом коммите или pull request Git-система отправляет webhook:

```
POST /api/integrations/git/webhook
{
  "project_id": "uuid",
  "owner_team_id": "uuid",
  "repo_url": "https://github.com/org/repo",
  "commit_sha": "abc123def456...",
  "event_type": "push",    // push | pull_request | release
  "metadata": {
    "branch": "main",
    "author": "student@example.com",
    "files_changed": 12
  }
}
```

### 16.2 Результат обработки

Система:
1. Вычисляет `content_hash` из `sha256(repo_url + commit_sha)`
2. Создаёт артефакт типа `code` с `git_url`
3. Возвращает `{ artifact_id, content_hash }`

Это позволяет **автоматически** создавать артефакты при каждом коммите без ручного ввода.

## 17. Управление доступом партнёров

### 17.1 Иерархия доступа

```
Администратор
    │
    ├─ Добавляет партнёрскую организацию (вуз/технопарк/гос.)
    ├─ Выдаёт партнёру email + пароль
    │
    ▼
Партнёр (логин в кабинет партнёра)
    │
    ├─ Добавляет компании (тип organization = "company")
    ├─ Выдаёт компании email + пароль
    ├─ Регистрирует менторов
    ├─ (если вуз) — верифицирует студентов
    │
    ▼
Компания (логин в кабинет компании)
    │
    └─ Создаёт задачи, нанимает исполнителей
```

### 17.2 Хранение учётных данных партнёров

MVP хранит пароли партнёров и компаний в `localStorage` браузера (не в БД):

- `sb:partner_creds` — {email: {passwordHash, orgId, orgName, orgType}}
- `sb:company_creds` — {email: {passwordHash, companyId, companyName, addedByEmail}}

Пароль хешируется через SHA-256 в браузере перед сохранением. Это решение подходит для MVP; в production потребуется серверная авторизация.

### 17.3 Учётные данные менторов

Менторы хранятся в `sb:mentor_registry` (localStorage):
```json
{
  "userId": "uuid",
  "email": "mentor@example.ru",
  "name": "Иван Иванов",
  "assignedBy": "partner@university.ru",
  "assignedAt": "2026-05-01T10:00:00Z",
  "passwordHash": "sha256..."
}
```

Ментора может зарегистрировать партнёр (вуз) или компания.

## 18. Пользовательские пути (User Journeys)

### 18.1 Путь студента

```
Регистрация → Профиль → Compliance → Верификация → Задачи → Отклик → Назначение → Работа → Артефакты → Навыки
```

**Детальный путь:**

1. **Регистрация**  
   `/login` → выбирает роль `student`, вводит email + пароль  
   `POST /api/auth/register` → получает `AuthUser` с ролью

2. **Заполнение профиля**  
   `/profile` → вводит имя  
   Документы: ставит галочку 152-ФЗ, выбирает НПД-статус  
   `PUT /api/compliance/profiles` → сохраняет согласия

3. **Верификация в вузе**  
   `/profile` → секция «Верификация студента вузом-партнёром»  
   Выбирает вуз из списка, вводит номер студбилета  
   `POST /api/verifications/students` → статус `pending`  
   Ждёт решения партнёра

4. **Поиск задачи**  
   `/tasks` → просматривает список, фильтрует по статусу  
   `/tasks/{id}` → изучает описание, бюджет, требуемые навыки

5. **Отклик**  
   Форма «Откликнуться» → вводит предложенную ставку и сообщение  
   `POST /api/tasks/{id}/apply` → отклик создан

6. **Ожидание назначения**  
   Студент ждёт, пока клиент просмотрит отклики и назначит его

7. **Выполнение задачи**  
   Работает над задачей, взаимодействует с ментором

8. **Сдача результата**  
   Загружает артефакты через `POST /api/artifacts`  
   Или через Git-интеграцию (автоматически)

9. **Получение оценки**  
   Ментор ставит `mentorship_seal = true` на артефакты  
   Клиент ставит `client_score`  
   Граф компетенций обновляется

---

### 18.2 Путь клиента/компании

```
Регистрация → Создание задачи → Просмотр откликов → Назначение → Сопровождение → Приёмка → Оценка
```

**Детальный путь:**

1. **Регистрация/вход**  
   Компания получает логин от партнёра (или регистрируется как `client`)

2. **Создание задачи**  
   `/tasks/new` → заполняет форму  
   `POST /api/tasks` → задача в статусе `open`

3. **Получение откликов**  
   Студенты сами находят задачу и откликаются

4. **Просмотр кандидатов**  
   `/tasks/{id}` → видит отклики с бейджами верификации вузов  
   Оценивает опыт, ставку, сопроводительное письмо

5. **Назначение исполнителя**  
   Выбирает ментора (если это первые 3 задачи с данным исполнителем — обязательно)  
   `POST /api/tasks/{id}/assign` → задача переходит в `assigned`

6. **Сопровождение**  
   Следит за прогрессом, общается с ментором

7. **Приёмка и оценка**  
   `POST /api/tasks/{id}/status` с `completed`  
   `POST /api/scoring/projects` с оценками

---

### 18.3 Путь ментора

```
Регистрация (партнёром) → Вход → Просмотр задач → Сопровождение → Логирование активности → Постановка печати → Обновление навыков
```

**Детальный путь:**

1. **Регистрация**  
   Партнёр создаёт учётную запись ментора через кабинет партнёра  
   `POST /api/users` + запись в `sb:mentor_registry`

2. **Вход**  
   `/login` → вводит email/пароль, выбирает роль `mentor`

3. **Просмотр назначенных задач**  
   `/tasks` → видит задачи, где указан как ментор

4. **Логирование активности**  
   Через страницу Admin (`/admin`) или API  
   `POST /api/mentor-activity` с типом действия, продолжительностью, весом

5. **Проверка артефактов**  
   `PATCH /api/artifacts/{id}` с `mentorship_seal: true`

6. **Обновление навыков исполнителя**  
   `PUT /api/capability` с уровнем владения и ссылкой на артефакт

7. **Финальная оценка проекта**  
   `POST /api/scoring/projects` с `mentor_score`

---

### 18.4 Путь администратора

```
Вход → Кабинет → Добавление партнёров → Выдача доступов → Мониторинг → Разбор диспутов
```

**Детальный путь:**

1. **Вход** с ролью `admin`

2. **Кабинет администратора** `/cabinet`  
   Видит статистику: пользователи, задачи, вузы, технопарки

3. **Добавление партнёра**  
   Форма «Добавить партнёра» → выбирает тип, вводит название  
   `POST /api/organizations` → организация создана

4. **Одобрение партнёра**  
   Кнопка «Одобрить» → статус меняется на `approved` (localStorage)

5. **Выдача доступа**  
   Иконка ключа → форма с email и паролем  
   Сохраняется в `sb:partner_creds`  
   Партнёр получает учётные данные и может войти

6. **Мониторинг**  
   Дашборд: открытые задачи, диспуты

7. **Разбор диспутов**  
   Находит задачи в `disputed`  
   `POST /api/tasks/{id}/status` с нужным статусом

---

### 18.5 Путь партнёра (вуза)

```
Вход → Кабинет партнёра → Добавление компаний → Регистрация менторов → Верификация студентов → Выдача доступов компаниям
```

**Детальный путь:**

1. **Вход** с email/паролем, выданным администратором  
   Роль: `partner`

2. **Кабинет партнёра** `/partner-cabinet`  
   Видит свою организацию (вуз)

3. **Добавление компаний**  
   Вводит название компании  
   `POST /api/organizations` с `type: "company"`  
   Сохраняет в `sb:company_creds`

4. **Выдача доступа компании**  
   Иконка ключа → задаёт email и пароль для компании

5. **Регистрация менторов**  
   Форма: имя, email, пароль ментора  
   `POST /api/users` → пользователь создаётся  
   Ментор добавляется в `sb:mentor_registry`

6. **Верификация студентов** (только для вузов)  
   Видит список заявок (`GET /api/verifications/university/{orgId}`)  
   Кнопки «Подтвердить» / «Отклонить»  
   `PATCH /api/verifications/{id}/review`

## 19. Все возможные сценарии по ролям

### 19.1 Сценарии для студента

| Сценарий | Описание | API |
|---|---|---|
| Первая регистрация | Студент создаёт аккаунт | `POST /api/auth/register` |
| Заполнение профиля | Студент указывает согласия и НПД | `PUT /api/compliance/profiles` |
| Верификация в вузе | Подача заявки на верификацию | `POST /api/verifications/students` |
| Поиск задачи | Просмотр открытых задач | `GET /api/tasks?status=open` |
| Отклик на задачу | Отправка заявки с предложением | `POST /api/tasks/{id}/apply` |
| Повторный отклик | Обновление существующей заявки | `POST /api/tasks/{id}/apply` (upsert) |
| Просмотр статуса | Задача уже назначена — student видит assignment | `GET /api/tasks/{id}/assignment` |
| Открытие диспута | Студент недоволен оценкой или условиями | `POST /api/tasks/{id}/disputes` |
| Просмотр навыков | Студент видит свой граф компетенций | `GET /api/capability?entity_type=user&entity_id={id}` |

**Сценарий: Студент не может получить назначение**

Причины и что делать:
- `pdn_consent = false` → Профиль → поставить галочку 152-ФЗ
- `npd_status = 'unknown'` → Профиль → выбрать статус НПД
- Нет откликов / клиент выбрал другого → ждать других задач
- Задача уже `assigned` → подать отклик нельзя (форма скрыта)

---

### 19.2 Сценарии для клиента

| Сценарий | Описание | Особенности |
|---|---|---|
| Создание задачи | Публикация задания для студентов | Только `client`, `admin`, `company` |
| Просмотр откликов | Оценка кандидатов с верификациями вузов | Видно только клиенту |
| Назначение с ментором | Первые 3 задачи с исполнителем | Обязательно для anti-disintermediation |
| Назначение без ментора | После 3 задач | Ментор опционален |
| Блокировка от compliance | 152-ФЗ или НПД не оформлены | Ошибка 409, подсказка в UI |
| Завершение задачи | Перевод в `completed` | Счётчик anti-disintermediation +1 |
| Отмена задачи | Перевод в `canceled` | Возможно из любого статуса |
| Открытие диспута | Клиент недоволен результатом | SLA 48ч |
| Оценка проекта | Выставление `client_score` | Влияет на `success_rate` с весом 0.30 |

**Сценарий: Клиент хочет повторно нанять того же студента**

После 3 успешных задач:
- `AntiDisintermediationRule.is_active = false`
- При назначении 4й задачи блокировка по ментору снята
- Клиент может указать ментора опционально или не указывать

## 20. Граничные случаи и защитная логика

### 20.1 Дублирование откликов

**Проблема:** студент случайно нажимает «Откликнуться» дважды.  
**Решение:** `INSERT ... ON CONFLICT DO UPDATE` по `(task_id, applicant_id)`. Дубликата не будет — обновится существующий отклик.

### 20.2 Задача уже назначена

**Проблема:** клиент пытается назначить другого исполнителя, когда задача уже `assigned`.  
**Решение:** при повторном `POST /{task_id}/assign` существующий `TaskAssignment` обновляется (`executor_id` и `mentor_id` заменяются), статус остаётся `assigned`.

### 20.3 Attempt to bypass anti-disintermediation

**Проблема:** клиент пытается назначить без ментора на первые задачи.  
**Решение:** жёсткая проверка в API: 409 с понятным сообщением. Обойти невозможно через UI — фронтенд дублирует логику, но даже прямой API-запрос будет отклонён.

### 20.4 Compliance не заполнен

**Проблема:** исполнитель не заполнил согласия.  
**Решение:** API блокирует назначение. Фронтенд показывает подсказку со ссылкой на Профиль. Студент исправляет за 10 секунд.

### 20.5 Навык без артефакта

**Проблема:** кто-то пытается добавить навык без доказательства.  
**Решение:** поле `evidence_artifact_id` — NOT NULL на уровне БД. API вернёт 422 без артефакта.

### 20.6 Диспут на завершённой задаче

**Проблема:** задача уже `completed`, но клиент обнаружил проблему позже.  
**Решение:** диспут можно открыть в любом статусе. Если задача `completed` — статус не меняется на `disputed` (условие: `if task.status not in {"disputed", "completed", "canceled"}`).

### 20.7 Повторная попытка верификации

**Проблема:** студент подал заявку в вуз и хочет подать ещё раз, не дожидаясь ответа.  
**Решение:** кнопка деактивируется если есть заявка в `pending` в этот же вуз (`hasPending` проверяется на фронтенде, повторных проверок в API нет — MVP).

### 20.8 Партнёр не найден при входе в кабинет

**Проблема:** партнёр вошёл, но его email не найден в `sb:partner_creds`.  
**Решение:** кабинет партнёра показывает «Данные не найдены. Обратитесь к администратору». Все функции недоступны.

### 20.9 Отсутствие менторов в списке

**Проблема:** клиент назначает задачу, но список менторов пуст.  
**Решение:** UI показывает: «Нет менторов — попросите вуз-партнёр назначить их в кабинете партнёра». Кнопка «+ Добавить ментора» позволяет выбрать любого пользователя системы и добавить его в реестр прямо в форме назначения.

### 20.10 Success rate при неполных оценках

**Проблема:** задача завершена, но ментор не поставил оценку.  
**Решение:** `compute_success_rate` нормирует на присутствующие оценки. Результат будет, но менее точный. Функция возвращает `None` только если вообще нет ни одной оценки.

## Глоссарий

| Термин | Определение |
|---|---|
| **Артефакт** | Неизменяемый результат работы с криптографическим хэшем |
| **Граф компетенций** | База навыков, подкреплённых артефактами |
| **PoM** | Proof of Mentorship — доказанная менторская активность |
| **360° Score** | Взвешенная оценка проекта из 4 источников |
| **Anti-disintermediation** | Защита от обхода платформы при прямом найме |
| **Compliance** | Профиль согласий (152-ФЗ на ПДн, 422-ФЗ на НПД) |
| **НПД** | Налог на профессиональный доход (самозанятые) |
| **SLA** | Service Level Agreement — гарантированное время ответа |
| **Mentorship Seal** | Одобрение артефакта ментором |
| **Reusability Index** | Индекс переиспользования артефакта (0–1) |
| **Upsert** | Вставка с обновлением при конфликте |
| **content_hash** | SHA-256 хэш содержимого артефакта |
