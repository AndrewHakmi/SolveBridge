# Соответствие платформы BMC (Tatarstan IT Platform)

Этот документ фиксирует, как элементы Business Model Canvas из `BMC_IT_Platform_Tatarstan-2 (2).pdf` отражаются в продуктовых модулях и данных платформы.

## Ключевые партнёры → Модули платформы

- **Вузы (КФУ ИТИС / Иннополис / КНИТУ‑КАИ ИКТЗИ)** → `organizations(type=university)` + `student_verifications` (привязка студента к вузу, статус верификации).
- **Минцифры РТ / IT‑парк / ОЭЗ Иннополис** → `organizations(type=government|infrastructure)` (реестр партнёров/интеграций; витрина/отчётность в будущем).
- **Платёжный провайдер (ЮKassa/Тинькофф)** → `payment_intents` (инициация, статус, external_id) + последующая интеграция.
- **«Мой налог» / самозанятые / юр.сопровождение** → `compliance_profile` (статусы НПД/152‑ФЗ/документы) + `invoices` (единый закрывающий документ — будущий модуль).

## Ключевые активности → Доменные процессы

- **Матчинг исполнителя и задачи** → `tasks` + `task_applications` + `task_assignments` (MVP); позже — ML‑скоринг/эмбеддинги.
- **Онбординг и верификация студентов** → `student_verifications` (статусы, документы, вуз‑партнёр).
- **Поддержка ментора клиента** → `task_assignments.mentor_id` + `mentor_activity_logs` (PoM).
- **Рейтинги и сообщество** → `ratings` (MVP); позже — агрегаты и витрины.
- **Диспуты (SLA 48ч), гарантия переделки** → `disputes` (deadline, статус, причина) + workflow задач.
- **Антифрод и защита от disintermediation** → `anti_disintermediation_rules` (MVP правила на пары компания‑исполнитель/первые N задач) + события контроля.
- **Compliance: 422‑ФЗ, 152‑ФЗ** → `compliance_profile` (MVP статусы), журналирование действий и доступов (позже).

## Ценностное предложение → Функции и UI

- **Для клиента**: экономия, гарантия, документы, “воронка найма” → `tasks`, `disputes`, `ratings`, `service_plans`, `subscription` + UI “постановка задачи/контроль/история”.
- **Для исполнителя**: оплачиваемые задачи + рост ставки по рейтингу → `task_applications`, `ratings`, `capability_graph`.
- **Что не обещаем**: не senior‑задачи/не доступ к продакшену → ограничения в `tasks.category/complexity`, флаги доступа, шаблоны условий.

## Отношения с клиентами (3 уровня сервиса) → Тарифы и SLA

- **Self‑Service / Managed / Enterprise** → `service_plans` + `company_subscriptions`.
- **SLA** → поля плана (`sla_minutes`, лимиты), вычисление дедлайнов (в т.ч. для диспутов).
- **Co‑creation (ментор клиента)** → закрепление ментора на задаче/компании, PoM‑сигналы.

## Потребительские сегменты → Роли и доступы

- **Клиенты (CTO/Lead/PM)** → `users.role=client`, `organizations(type=company)`.
- **Исполнители (студенты)** → `users.role=student`, верификация и профиль компетенций.
- **Менторы** → `users.role=mentor`, PoM‑логирование.
- **Ops/Admin** → `users.role=admin` (управление справочниками/партнёрами/модерация).

## Каналы сбыта → Артефакты маркетинга (в продукте)

- Лендинг/калькулятор, демо‑консультация → отдельный web‑маркетинг слой (вне текущего MVP UI).
- Личный кабинет Self‑Service / Managed+ → текущий web‑app UI (frontend).

## Структура издержек и потоки доходов → Биллинг (MVP)

- Комиссия, подписки, премиум‑сервисы → `payment_intents` + `service_plans/subscriptions` (MVP учет), дальше — счета/акты/эквайринг.
