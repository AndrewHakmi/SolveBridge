# Соответствие платформы BMC (канвас Татарстан IT Platform)

Источник: `.trae/documents/BMC_IT_Platform_Tatarstan-2 (2).pdf`.

Цель этого файла — показать, что в продукте **реально есть** для каждого блока канваса (B2C для студентов + B2B/B2G для технопарков/Минцифры), и где остаются «дыры» для следующей итерации.

## 1) Ключевые партнёры

| Блок канваса | Как отражено в платформе | Статус |
|---|---|---|
| Вузы (КФУ ИТИС / Иннополис / КНИТУ‑КАИ) | `organizations(type=university)` + `student_verifications` (MVP) | MVP (есть модель/эндпоинты, нет полноценного workflow/ролей) |
| Минцифры / технопарки / инфраструктура | `organizations(type=government|infrastructure)` + UI «Панель партнёра» | MVP (есть витрина и метрики) |
| Платёжный провайдер (ЮKassa/Тинькофф) | `payment_intents` | MVP (учёт намерений, нет интеграции) |
| «Мой налог» / самозанятые / юр.сопровождение | поле под compliance заложено в BMC, в продукте пока нет доменной сущности | Gap |

## 2) Ключевые активности

| Активность | Как отражено в платформе | Статус |
|---|---|---|
| Матчинг исполнителя и задачи | `tasks` + `task_applications` + `task_assignments` | MVP (ручной, без ML-скоринга) |
| Онбординг и верификация студентов | `student_verifications` (`/api/verifications/students`) | MVP |
| Поддержка ментора клиента | `mentor_activity_logs` + роль mentor + экран «Инструменты» | MVP (ручной ввод активностей) |
| Управление рейтингами и сообществом | `ratings` (`/api/ratings`) | MVP (есть записи, нет агрегатов/витрин) |
| Диспуты (SLA 48ч), гарантия переделки | `disputes` (`/api/tasks/{id}/disputes`) | MVP (есть диспут и SLA-таймер, нет полного процесса решения) |
| Антифрод / защита от disintermediation | `anti_disintermediation_rules` | Gap (таблица есть, логика не включена) |
| Compliance (422‑ФЗ, 152‑ФЗ) | Логически учтено в BMC, но в продукте пока нет `compliance_profile` и политик доступа | Gap |

## 3) Ценностное предложение

| Для кого | Как демонстрируем в продукте сейчас | Статус |
|---|---|---|
| Клиент: дешевле, быстрее, с гарантией | «Задачи» + назначение + диспут (SLA) | MVP |
| Клиент: инфраструктура для ментора | роль mentor + логи PoM | MVP |
| Клиент: юр. защита/закрывающие документы | модель не реализована | Gap |
| Исполнитель: оплачиваемые задачи + рост ставки | «Задачи» + отклик + портфолио (артефакты) | MVP |
| Исполнитель: менторство | mentor + PoM логи | MVP |

## 4) Отношения с клиентами (Self / Managed / Enterprise)

| Уровень | Как отражено в платформе | Статус |
|---|---|---|
| Self-Service | `service_plans(self)` + UI показ планов | MVP |
| Managed | `service_plans(managed)` + `company_subscriptions` | MVP (учёт; SLA ещё не привязан к задачам автоматически) |
| Enterprise | `service_plans(enterprise)` | MVP (учёт; кастомизация/интеграции — Gap) |

## 5) Потребительские сегменты и роли

| Сегмент | Как отражено в продукте | Статус |
|---|---|---|
| Студенты (B2C) | роль `student`, сценарий: задачи → отклик | MVP |
| Клиенты (B2B) | роль `client`, сценарий: создать задачу → назначить | MVP |
| Менторы | роль `mentor`, PoM логи | MVP |
| Технопарки/Минцифры (B2B/B2G) | роль `admin`, «Панель партнёра» | MVP |

## 6) Каналы сбыта

Слой каналов (лендинг, outreach, телеграм‑комьюнити) как правило живёт **вне** web‑app. Внутри продукта мы закрываем только «личный кабинет и процесс оказания услуги».

| Канал | Как отражено в продукте | Статус |
|---|---|---|
| Self-Service кабинет | UI + «Задачи» | MVP |
| Managed/Enterprise кабинет | планы/подписки заложены | MVP/GAP (нужно развивать CS-процессы и SLA) |

## 7) Издержки и доходы (биллинг)

| Блок | Как отражено в продукте | Статус |
|---|---|---|
| Комиссия и транзакции | `payment_intents` | MVP (без оплаты) |
| Подписки | `service_plans` + `company_subscriptions` | MVP (учёт) |
| Премиум-сервисы (срочность, супер-ментор, NDA, API) | пока нет модели | Gap |
| B2B аналитика для вузов/Минцифры | «Панель партнёра» (сейчас только базовые метрики) | MVP |

## 8) Что уже можно показать первым пользователям (демо)

- **Студентам (B2C):** «Задачи» → открыть задачу → «Откликнуться».
- **Клиентам (B2B):** «Создать задачу» → открыть задачу → назначить исполнителя.
- **Технопарку/Минцифры (B2G/B2B):** «Ещё» → «Панель партнёра» → метрики + партнёры + уровни сервиса.

## 9) Следующая итерация (чтобы канвас был “закрыт” по-настоящему)

1. **Compliance:** `compliance_profile` + минимальные статусы (НПД/152‑ФЗ/документы) и ограничения доступа.
2. **Anti‑disintermediation:** включить правила «первые N задач через платформу» (на уровне workflow задач + алерты).
3. **Managed SLA:** привязать SLA задач/диспутов к `company_subscriptions.plan_code` автоматически.
4. **Биллинг:** связать `payment_intents` с провайдером (песочница) и статусами задач.
